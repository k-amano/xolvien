# Xolvien — 改修要件

## 1. 実装方針

### 1.1 実装の優先順位

以下の順序で実装を進める。

1. テスト実行 + 自動修正ループ（単体テスト → 結合テスト → E2Eテストの順に段階実装）
2. PR自動作成
3. GitHub Issue連携（Webhook → タスク自動作成・実行）

マルチユーザー対応（ストリーミングのブロッキング解消・GitHub OAuth認証等）は、シングルユーザーでの全機能実装完了後に着手する。

---

## 2. 実行フロー設計

### 2.1 基本方針

- ステップごとにユーザーの確認を挟む（意図と異なる方向への進行を早期に防ぐ）
- コミットは各ステップ完了時に都度行う（全完了まで待たない）
- pushは最終確認後に一括で行う
- テストコードはClaudeが自動生成・修正する（ユーザーがテストコマンドを入力する必要はない）

### 2.2 新規作成のフロー

```
1. 指示入力
2. 要件確認（Claude ↔ ユーザー）
3. プロンプト確認 → ユーザーが承認
4. Claude が実装コード + 単体テストを生成
5. 単体テスト実行 → 失敗なら自動修正ループ（最大3回）
6. ユーザーが実装を確認 → 承認 or 差し戻し
   → 承認 → 自動コミット（実装 + 単体テスト）
   → 差し戻し → ステップ3に戻り指示を修正
7. Claude が結合テストを生成・実行 → 失敗なら自動修正ループ
   → 成功 → 自動コミット（結合テスト）
8. Claude がE2Eテストを生成・実行 → 失敗なら自動修正ループ
   → 成功 → 自動コミット（E2Eテスト）
9. 全テスト成功 → ユーザーが最終確認（テスト結果・スクリーンショット）
   → 承認 → push → PR作成
   → 差し戻し → ステップ6に戻り追加指示
```

### 2.3 既存コード修正のフロー

```
1. 指示入力
2. 要件確認（Claude ↔ ユーザー）
3. プロンプト確認 → ユーザーが承認
4. Claude が既存テストを確認した上で実装コードを修正
   + 必要に応じてテストコードも修正・追加
5. 単体テスト実行（既存 + 新規）→ 失敗なら自動修正ループ
6. ユーザーが実装を確認 → 承認 or 差し戻し
   → 承認 → 自動コミット
   → 差し戻し → ステップ3に戻り指示を修正
7. 結合テスト実行（既存 + 新規）→ 失敗なら自動修正ループ
   → 成功 → 自動コミット
8. E2Eテスト実行（既存 + 新規）→ 失敗なら自動修正ループ
   → 成功 → 自動コミット
9. 全テスト成功 → ユーザーが最終確認
   → 承認 → push → PR作成
   → 差し戻し → ステップ6に戻り追加指示
```

新規作成との違い：
- ステップ4: テストを「生成する」ではなく「既存を確認した上で修正・追加する」
- ステップ5・7・8: 既存テストも含めて実行し、デグレード（既存機能の破壊）も検出する

### 2.4 ユーザーの確認ポイント

| タイミング | 確認内容 | 承認後の動作 | 差し戻し時の動作 |
|---|---|---|---|
| ステップ3 | プロンプトが意図通りか | 実装開始 | 指示を修正して再生成 |
| ステップ6 | 実装が意図通りか（変更ファイル一覧・diff表示） | コミット → テスト続行 | ステップ3に戻り指示を修正 |
| ステップ9 | テスト結果・動作が意図通りか | push → PR作成 | ステップ6に戻り追加指示 |

## 3. テスト設計

### 3.1 テストコードの生成方針

- Claudeが実装コードと同時にテストコードを生成する
- ユーザーがテストコマンドを入力する必要はない
- テストコマンドはClaudeがリポジトリの構成（package.json、pyproject.toml等）から自動判断する
- 新規作成時はテストコードを新規生成、既存修正時は既存テストを確認した上で修正・追加する

### 3.2 テスト実行の段階的実装計画

要件書（requirements (8).md）に定義された3層構成を以下の順序で実装する。

| フェーズ | 内容 |
|---|---|
| フェーズ1 | 単体テスト（依存なし、高速） |
| フェーズ2 | 結合テスト（DB・API接続が必要） |
| フェーズ3 | E2Eテスト（Playwright、ブラウザ操作） |

### 3.3 自動修正ループ

- 最大リトライ回数: 3回
- 失敗時のフィードバック内容: 失敗したテスト名・エラーメッセージ・標準出力
- E2Eテスト失敗時はスクリーンショットもフィードバックに含める
- リトライ上限到達時はユーザーに通知して手動対応を促す
